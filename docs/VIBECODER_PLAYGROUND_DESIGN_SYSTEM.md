# VibeCoder Playground - Design Masterplan (MVP v1)

## 문서 목적
- 이 문서는 `바이브코더들의 놀이터` 홈페이지를 디자인 중심으로 설계하고, 이후 구현/운영 과정에서 반복 학습할 수 있도록 만든 기준 문서다.
- 범위는 브랜드 톤, 화면 구조, UX 플로우, 컴포넌트, 댓글/피드백 경험, FastAPI 연동 요구사항까지 포함한다.
- 프론트 제작은 Google Stitch 아이디어 정리 -> v0 구현, 백엔드는 FastAPI 기반으로 안티그래비티 IDE에서 확장하는 흐름을 전제로 한다.

## 1) 제품 한 줄 정의
"바이브코더가 만든 결과물과 실험을 자랑하고, 서로의 바이브를 댓글로 피드백하는 커뮤니티 쇼케이스."

## 3) 브랜드/비주얼 방향 (Design North Star)
### 톤 앤 매너
- 키워드: Playful, Experimental, Electric, Handmade
- 메시지: "완성도보다 바이브. 실험도 작품이다."
- 금지: 지나치게 기업형, 딱딱한 SaaS 템플릿 분위기

### 시각 컨셉
- 배경: 다층 그라디언트 + 아주 약한 노이즈 텍스처
- 카드: 유리 느낌보다 "포스터 스택"(살짝 겹침/기울기) 스타일
- 강조 요소: 스티커형 라벨(New, Hot, Weird, WIP)
- 모션: 입장 스태거, 카드 호버 시 미세한 틸트, 좋아요/댓글 반응 이펙트

### 컬러 시스템 (초안)
- `--bg-0`: #0B1020 (딥 네이비)
- `--bg-1`: #111936
- `--card`: #161F42
- `--text-0`: #F4F7FF
- `--text-1`: #B8C3E6
- `--accent-0`: #23D5AB (민트)
- `--accent-1`: #FFB547 (앰버)
- `--accent-2`: #FF5D8F (코랄 핑크)
- `--danger`: #FF6B6B

원칙:
- 포인트 컬러는 한 화면에 최대 2개만 적극 사용한다.
- CTA는 민트(`--accent-0`) 고정으로 일관성 유지.
- 상태 메시지는 색 + 아이콘 + 텍스트를 함께 사용해 접근성을 확보.

### 타이포그래피
- 디스플레이: `Space Grotesk` (영문 제목/숫자)
- 본문: `Pretendard` (한글 가독성)
- 코드/태그: `JetBrains Mono`

타입 스케일:
- Hero Title 56/64
- H1 40/48
- H2 32/40
- H3 24/32
- Body 16/26
- Caption 13/20

## 4) 정보구조(IA)
### 최상위 메뉴
- Home (피드)
- Explore (태그/필터 중심 탐색)
- Submit (작품 등록)
- Challenges (향후 확장, 초기엔 비활성 가능)
- About Book (책 소개/출간 연동)

### 핵심 페이지 (MVP)
1. 홈/피드
2. 작품 상세
3. 작품 등록
4. 프로필(내가 올린 작품, 내가 남긴 댓글)
5. 관리자(댓글 신고/블라인드)

## 5) 화면별 설계
## 5.1 홈/피드
목표:
- 첫 방문 5초 내에 "여긴 자랑하는 놀이터"라는 인상을 준다.

구성:
- Hero: 강한 메시지 + "작품 올리기" CTA
- Trending Strip: 이번 주 반응 좋은 작품 6개
- Main Feed: 카드 그리드 (최신/인기 탭)
- Filter Bar: 플랫폼(웹/앱/툴), 스택, 태그

카드 최소 정보:
- 썸네일
- 제목
- 한 줄 설명
- 태그 2-3개
- 작성자 닉네임
- 좋아요 수 / 댓글 수

## 5.2 작품 상세
목표:
- 작품 감상 -> 반응(좋아요/댓글) -> 창작자 팔로업 링크 클릭의 흐름 강화

구성:
- 상단: 제목, 작성자, 날짜, 태그
- 미디어 영역: 이미지 캐러셀 또는 임베드
- 본문: 제작 의도, 사용 기술, 느낀 점
- 액션: Like, 공유, 원문/배포 링크
- 댓글 섹션: 정렬(최신/공감순), 신고, 작성 입력창

댓글 UX 원칙:
- 첫 댓글 유도 문구 제공("좋았던 포인트 한 가지를 남겨보세요")
- 작성 전 가이드 1줄 노출("존중 기반 피드백만 허용")
- 악성 징후 단어 포함 시 제출 전 소프트 경고

## 5.3 작품 등록
목표:
- 1분 업로드를 달성한다.

폼 항목 (MVP):
- 제목 (필수)
- 한 줄 소개 (필수)
- 상세 설명 (선택)
- 미리보기 이미지 URL 또는 업로드
- 데모 링크, GitHub 링크
- 태그(최대 5)
- 플랫폼 선택(웹/앱/AI/기타)

UX 세부:
- 저장 전 실시간 미리보기 카드 제공
- 필수값 누락 시 상단 요약 에러 + 필드 인라인 에러 동시 제공
- 업로드 성공 후 "내 작품 보기" 바로가기 제공

## 5.4 프로필
구성:
- 프로필 헤더(닉네임, 소개)
- 탭: 작품 / 댓글 / 좋아요
- 사용자 활동 점수(가벼운 게이미피케이션)

## 5.5 관리자
구성:
- 신고 큐(최신순)
- 댓글 블라인드/복구
- 사용자 24시간 제한
- 금칙어 관리(향후)

## 6) 컴포넌트 시스템 (v0 구현 가이드)
필수 컴포넌트:
- `TopNav`
- `HeroBanner`
- `FilterChips`
- `ProjectCard`
- `ProjectMeta`
- `CommentList`
- `CommentComposer`
- `Toast`
- `ReportModal`

컴포넌트 원칙:
- 상태를 명확히 구분: default/hover/active/loading/disabled/error
- 카드와 댓글 리스트는 스켈레톤 로딩 상태를 기본 제공
- 모바일 1열, 태블릿 2열, 데스크톱 3-4열 그리드

## 7) 상호작용/모션 규칙
- 페이지 진입: 240-320ms, ease-out
- 카드 스태거: 30-50ms 간격
- 버튼 피드백: 120ms 이내
- 과한 패럴랙스/무거운 애니메이션 금지 (성능 우선)

## 8) 접근성/국제화 최소 기준
- 본문 대비율 WCAG AA 이상
- 키보드 포커스 링 항상 표시
- 모든 이미지에 대체 텍스트
- 댓글 작성/오류 메시지 스크린리더 라벨링
- 기본 언어 한국어, 추후 영문 토글 확장 가능하도록 문자열 분리

## 9) 데이터 모델/백엔드 연계 요구사항 (FastAPI)
핵심 엔티티:
- User
- Project
- ProjectTag
- Comment
- Reaction(Like)
- Report

### 9.1 엔티티 필드 기준 (MVP)
User:
- `id` (uuid)
- `email` (unique, nullable 허용 가능)
- `nickname` (unique)
- `bio` (nullable)
- `avatar_url` (nullable)
- `role` (`user`, `moderator`, `admin`)
- `created_at`, `updated_at`

Project:
- `id` (uuid)
- `author_id` (fk -> user)
- `title`
- `summary`
- `description` (nullable)
- `thumbnail_url` (nullable)
- `demo_url` (nullable)
- `repo_url` (nullable)
- `platform` (`web`, `app`, `ai`, `etc`)
- `status` (`published`, `hidden`, `deleted`)
- `like_count`, `comment_count` (denormalized)
- `created_at`, `updated_at`

Comment:
- `id` (uuid)
- `project_id` (fk)
- `author_id` (fk)
- `parent_id` (nullable, 대댓글용)
- `content`
- `status` (`visible`, `hidden`, `deleted`)
- `created_at`, `updated_at`

Report:
- `id` (uuid)
- `target_type` (`project`, `comment`)
- `target_id`
- `reporter_id`
- `reason` (`spam`, `abuse`, `adult`, `other`)
- `memo` (nullable)
- `status` (`open`, `reviewing`, `resolved`, `rejected`)
- `created_at`, `resolved_at`

### 9.2 MVP API (초안)
- `GET /projects` (최신/인기/태그/플랫폼 필터)
- `POST /projects`
- `GET /projects/{id}`
- `POST /projects/{id}/like`
- `DELETE /projects/{id}/like`
- `GET /projects/{id}/comments`
- `POST /projects/{id}/comments`
- `POST /comments/{id}/report`
- `GET /me/projects`
- `GET /admin/reports`
- `PATCH /admin/reports/{id}` (처리상태 변경)

### 9.3 API 응답 원칙
- 리스트 응답은 페이지네이션 표준화: `items`, `next_cursor`
- 생성 API는 생성 리소스의 `id`와 핵심 메타를 즉시 반환
- 에러 응답은 `code`, `message`, `field_errors` 형태 고정
- 프론트 UX 동기화를 위해 `like_count`, `comment_count`는 상세/리스트에서 항상 제공

### 9.4 운영 안전장치
- 댓글 작성 레이트리밋 (예: 분당 N회)
- 기본 금칙어 필터
- 신고 누적 임계치 도달 시 자동 임시 숨김
- 동일 IP/계정의 반복 신고 스팸 방지
- 관리자 액션 로그 저장 (누가/언제/무엇을)

### 9.5 실시간 기능 우선순위
- P0: 폴링 기반 댓글 새로고침 (MVP)
- P1: WebSocket 기반 라이브 댓글/현재 참여자 수
- P2: 작성 중 표시(typing indicator)

## 10) 출시 단계 제안
Phase 1 (출간 전 2-3주):
- 홈/상세/등록/댓글/신고 MVP 오픈

Phase 2 (출간 직후):
- 주간 챌린지 + 배지
- 작품 큐레이션 섹션

Phase 3:
- 팀 프로젝트 카테고리
- API 공개 또는 위젯 임베드

## 11) 제작 워크플로우 (Google Stitch -> v0 -> FastAPI)
1. Google Stitch에서 화면 톤/무드보드/레이아웃 탐색
2. v0에서 컴포넌트 단위로 UI 생성 및 반응형 조정
3. v0 결과물을 코드로 Export하여 로컬 프로젝트에 반영
4. 안티그래비티 IDE에서 코드 정리/리팩터링 및 FastAPI 연동
5. GitHub 브랜치에 커밋 -> PR 생성 -> 리뷰/머지
6. main 반영 후 배포 환경 동기화

### 11.1 Stitch -> v0 핸드오프 프로세스 (권장)
0. 디자인 원칙 고정
- 이 문서의 색상/타입/컴포넌트 원칙을 먼저 확정하고 시작한다.

1. Google Stitch 단계 (탐색/디자인 결정)
- 화면별 레이아웃 2-3안 탐색: Home, Detail, Submit, Profile
- 최종안 선택 후 토큰 확정: 색상, 간격, 라운드, 그림자, 타이포 스케일
- 컴포넌트 인벤토리 생성: 카드, 필터칩, 댓글, 모달, 네비게이션

2. 핸드오프 패키지 생성 (Stitch 산출물 정리)
- 화면 캡처: 데스크톱/모바일 각각 1세트
- 컴포넌트 상태표: default/hover/active/loading/error
- 인터랙션 노트: 애니메이션 시간, 진입/전환 규칙
- 카피 가이드: Hero 메시지, 빈 상태 문구, 댓글 안내 문구

3. v0 구현 단계 (구조화된 이식)
- 먼저 레이아웃 골격만 생성
- 다음으로 핵심 컴포넌트 이식 (`ProjectCard`, `CommentComposer`, `ReportModal`)
- 마지막으로 반응형 및 접근성(포커스링/대비율/라벨) 보정

4. 디자인 QA (Stitch 원본 대비)
- 시각 일치율 체크: 색상/타입/간격/컴포넌트 상태
- UX 일치율 체크: CTA 위치, 댓글 작성 흐름, 신고 진입 동선
- 수정사항은 `IMPLEMENTATION_LEARNING_LOG.md`에 Session으로 기록

### 11.2 핵심 원칙
- Stitch는 "디자인 결정 도구", v0는 "구현 가속 도구"로 역할을 분리한다.
- v0에서 임의로 디자인을 바꾸지 않고, 변경 시 반드시 로그를 남긴다.
- 모바일 우선으로 검증한 뒤 데스크톱으로 확장한다.

### 11.3 v0 -> 안티그래비티 -> GitHub 저장 프로세스
1. v0 Export 반영
- v0에서 생성한 코드를 기능 단위로 분리해서 로컬 프로젝트에 붙인다.
- 붙인 직후 실행 확인(렌더링/반응형/기본 인터랙션) 먼저 통과시킨다.

2. 안티그래비티 IDE 수정
- 컴포넌트 구조 정리(파일 분리, props 정리, 중복 제거)
- API 연동 코드 추가(FastAPI endpoint 매핑)
- 에러/로딩/빈 상태를 디자인 규칙에 맞게 보강

3. Git 브랜치 전략
- 작업 시작: `feature/<scope>` 브랜치 생성
- 예시: `feature/feed-card-v1`, `feature/comment-report-flow`
- 하나의 PR은 하나의 사용자 가치(예: 댓글 신고 플로우 완성)만 담는다.

4. 커밋 규칙
- 커밋 메시지 형식: `type(scope): why`
- 예시: `feat(comment): add report flow to prevent abusive feedback`
- 디자인 수정 커밋에는 반드시 근거 섹션/로그 파일 업데이트를 포함한다.

5. PR 규칙
- PR 본문 필수 항목: 변경 요약, 스크린샷(모바일/데스크톱), 테스트 결과, 리스크
- 체크리스트: 접근성(포커스/대비), 반응형, API 에러 처리, 모더레이션 동선
- 머지 전 최소 1회 리뷰(셀프 리뷰 포함)로 UI 일관성 확인

6. 머지 후 동기화
- `main` 머지 후 배포 브랜치/환경 동기화
- 릴리즈 노트에 사용자 체감 변경점 기록
- 같은 날 `IMPLEMENTATION_LEARNING_LOG.md`에 결과/회고 반영

## 12) 학습용 체크리스트 (반복 학습 템플릿)
각 스프린트 종료 시 아래를 반드시 기록:
- 이번에 개선한 화면/컴포넌트
- 변경 이유(사용자 피드백 또는 지표)
- 잘 된 점 3개
- 아쉬운 점 3개
- 다음 실험 가설 1-2개

## 13) 리서치 반영 메모 (내부/외부)
### 내부 패턴 반영
- 커뮤니티 쇼케이스 IA는 `피드 -> 상세 -> 댓글 -> 신고` 흐름이 끊기지 않아야 재방문율이 오른다.
- 업로드 장벽을 낮출수록(1분 업로드) 초기 콘텐츠 풀 형성 속도가 빨라진다.
- 신고 UX는 숨겨진 관리자 기능이 아니라 사용자 여정 안에 자연스럽게 배치해야 한다.

### 외부 베스트 프랙티스 반영
- 2026년 트렌드는 과한 정제보다 "실험적/핸드메이드" 톤이 강세이며, 놀이터 컨셉과 정합성이 높다.
- 피드는 단순 좋아요 외에도 사용자 제어 신호(관심/비관심)를 주면 추천 품질이 빨리 안정화된다.
- 댓글/피드백은 사후 제재보다 작성 시점 가이드(소프트 인터벤션)가 품질 유지에 유리하다.

### 본 프로젝트에 적용한 결정
- 카드/라벨 중심 시각 언어를 유지하되, 텍스트 가독성과 정보 밀도를 우선한다.
- 신고/블라인드/제한 기능을 MVP에 포함한다(출시 후 추가 아님).
- 실시간 기능은 단계적으로 도입하고, 초기엔 안정성 중심으로 폴링을 채택한다.
